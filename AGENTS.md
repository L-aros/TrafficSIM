# AGENTS

本文件定义 AI 代理（如 Codex/Claude）在本仓库内的协作规范。

## 1. 目标
- 在不破坏现有行为的前提下，快速、可验证地完成需求。
- 优先修复缺陷与可维护性问题，其次再做样式或重构优化。

## 2. 工作方式
1. 先读上下文：
   - `README.md`
   - `DEVELOPMENT.md`
   - 相关源码与 `docs/`
2. 明确改动范围后再编辑，避免盲改。
3. 小步修改，保证每一步都可验证。
4. 完成后提供变更说明与验证结果。

## 3. 编辑原则
- 不修改与任务无关的文件。
- 不进行破坏性操作（如强制回滚、删除大量文件）除非用户明确要求。
- 保持现有项目结构与命名风格。
- 新增配置字段或行为时，补充默认值与容错逻辑。

## 4. Android/LSPosed 项目注意事项
- 作用域相关逻辑变更需要关注 `com.android.phone` / `android` 兼容性。
- 切卡逻辑调整需考虑：
  - 规则优先级（`priority`）
  - 冷却时间（`cooldownSec`）
  - 离开 Wi-Fi 策略（`leaveDelaySec`、`revertOnLeave`）
- Root 或广播链路失败时应保留降级或日志提示。

## 5. 验证要求
- 最少执行一次构建验证：
  - `./gradlew :app:assembleDebug`（Linux/macOS）
  - `./gradlew.bat :app:assembleDebug`（Windows）
- 若改动影响行为，需说明：
  - 触发条件
  - 预期结果
  - 实际结果（或未执行原因）

## 6. 文档同步
- 当出现以下变更时必须更新文档（`README.md` 或 `docs/`）：
  - 新增/删除用户可见功能
  - 配置项变更
  - 安装、权限或使用流程变更

## 7. 输出格式
- 结果应包含：
  - 改了什么
  - 为什么这样改
  - 如何验证
  - 未完成项与风险（如有）

## 8. 分支与提交约束
- 不允许直接向 `main` 提交。
- 默认在 `feature/*`、`fix/*`、`docs/*` 分支工作。
- 提交信息遵循 Conventional Commits：
  - `feat(scope): ...`
  - `fix(scope): ...`
  - `refactor(scope): ...`
  - `docs(scope): ...`

## 9. PR 质量门槛
- PR 描述必须包含：
  - 背景与目标
  - 关键改动点
  - 风险与回滚方案
- 行为改动必须包含验证记录：
  - 触发条件
  - 预期结果
  - 实际结果
- 未验证项必须明确标记为风险，不得隐去。

## 10. 禁止事项（严格）
- 禁止“顺手重构”无关模块。
- 禁止仅凭推测修改切卡核心逻辑。
- 禁止跳过构建验证后直接声明完成。
